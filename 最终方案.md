# 个人 AI 工作流助手 - 最终方案

## 一、最终定位

本项目不再定位为“完全本地、绝对隐私”的 AI 助手。这个方向虽然听起来安全，但在个人电脑无法运行高质量大模型的情况下，会牺牲智能效果，导致核心工作做不好。

最终定位调整为：

> 一个面向个人知识和工作文档的 AI 工作流助手，使用云端大模型保证智能效果，同时通过本地预处理、敏感信息脱敏、上传预览、执行日志和成本统计，让每一次 AI 调用都可控、可审计、可追踪。

一句话总结：

> 云端智力，本地边界。

本项目的差异化不再是“完全不上传云端”，而是“不会把用户文件黑盒式地丢给云端”。用户始终知道哪些内容会被发送、为什么发送、发送给哪个模型、预计消耗多少 token、最终花了多少钱。

---

## 二、核心差异化

如果只是做一个 LangChain Agent + Gradio + 几个 Python 工具，本项目很容易被 ChatGPT、Claude、Dify、Coze、LangFlow 等现成工具替代。

最终差异点应聚焦在两个方向：

### 2.1 可控上传

用户选择本地文件后，系统不是直接把全文发送给云端 LLM，而是先在本地完成：

- 文件解析
- 内容分块
- 关键信息抽取
- 敏感信息检测
- token 估算
- 成本预估
- 上传内容预览

然后由用户选择：

- 直接发送
- 脱敏后发送
- 只发送选中片段
- 取消任务

### 2.2 可审计 Agent

每次任务都生成完整 RunLog，记录：

- 用户原始请求
- 调用过哪些工具
- 每步输入和输出摘要
- 每步耗时
- 输入 token / 输出 token
- 估算费用
- 是否命中缓存
- 是否触发敏感信息拦截
- 最终结果保存路径

这让 Agent 从“黑盒聊天工具”变成“可追踪的工作流执行器”。

---

## 三、用户模式设计

不同任务对智能效果和隐私边界的要求不同，因此系统提供多种模式，而不是强行承诺一种绝对策略。

| 模式 | 适用场景 | 数据策略 |
|---|---|---|
| 快速模式 | 普通笔记、公开资料、低敏任务 | 直接调用云端 LLM，追求效果和速度 |
| 隐私增强模式 | 会议纪要、工作文档、合同初稿 | 本地预处理，只发送必要片段，可自动脱敏 |
| 手动确认模式 | 高敏感文件、客户资料、财务信息 | 发送前展示上传预览，用户确认后才发 |
| 本地兜底模式 | API 不可用或极敏感任务 | 只做本地解析、规则提取、模板生成，不承诺复杂推理质量 |

默认建议使用“隐私增强模式”，兼顾效果和边界。


## 七、v0.1 必做功能

v0.1 不追求功能多，而要追求闭环完整、安全边界清晰、演示有说服力。

### 7.1 文件处理闭环

- 读取本地 txt / md 文件
- 支持 pdf / docx 可作为加分项
- 提炼摘要、待办、风险点
- 保存为 Markdown
- 返回保存路径

### 7.2 上传预览

发送云端前展示：

- 即将发送的内容片段
- 是否包含敏感信息
- 预计输入 token
- 预计费用
- 使用模型
- 用户确认按钮

### 7.3 敏感信息检测与脱敏

v0.1 先用规则检测即可：

- 手机号
- 邮箱
- 身份证号
- API Key 常见格式
- 密码字段
- 绝对路径

脱敏示例：

```text
张三 → PERSON_1
13812345678 → PHONE_1
admin@example.com → EMAIL_1
sk-xxxx → API_KEY_1
```

### 7.4 安全边界

- 路径白名单
- 文件大小限制
- 禁止读取 `.env`、私钥、系统配置等敏感文件
- 写文件前检查目标目录
- 覆盖已有文件前要求确认
- 单次工具调用次数上限
- 单次 token / cost 上限

### 7.5 RunLog

v0.1 必须实现结构化日志：

```python
from dataclasses import dataclass, field
from typing import List

@dataclass
class StepLog:
    step_id: int
    name: str
    input_preview: str
    output_preview: str
    duration_ms: int
    tokens_in: int = 0
    tokens_out: int = 0
    cost_yuan: float = 0.0
    status: str = "success"

@dataclass
class RunLog:
    run_id: str
    user_query: str
    mode: str
    model: str
    steps: List[StepLog] = field(default_factory=list)
    total_tokens_in: int = 0
    total_tokens_out: int = 0
    total_cost_yuan: float = 0.0
    result_path: str | None = None
```

### 7.6 流式反馈

Agent 执行多步骤任务时，界面必须持续反馈：

- 正在解析文件
- 正在检测敏感信息
- 等待用户确认上传
- 正在调用模型
- 正在保存结果
- 已完成

## 九、缓存策略

v0.1 只做轻量缓存，不引入本地 embedding 模型。

### 9.1 文件级缓存

基于文件路径、修改时间、文件大小生成缓存 key。

作用：

- 避免重复读取同一文件
- 避免重复解析同一文件
- 降低本地 I/O 和预处理成本

### 9.2 上传片段缓存

同一文件、同一任务、同一模式下，如果上传片段未变化，可复用上次预处理结果。

### 9.3 暂不做语义缓存

语义缓存需要 embedding，会引入额外复杂度和依赖。对于面试版项目，文件级缓存 + 上传预览缓存已经足够。

## 十、成本控制

系统需要硬性熔断，避免 Agent 循环 bug 或大文件导致费用失控。

建议配置：

```yaml
limits:
  max_file_size_mb: 5
  max_tool_calls_per_request: 15
  max_tokens_per_request: 50000
  max_cost_per_request_yuan: 0.5
```

当预计费用超过阈值时：

1. 停止自动调用
2. 告知用户原因
3. 建议改为只发送部分片段
4. 允许用户手动确认继续

## 十三、面试讲法

### 13.1 项目价值

> 我没有做一个简单的 ChatGPT 套壳，而是做了一个“可控上传、可审计执行”的个人工作流助手。它承认云端大模型的智能优势，但不会把用户文件黑盒式地直接丢给云端。系统会先本地解析、检测敏感信息、估算 token 和成本，再让用户确认发送范围。

### 13.2 和现成工具的区别

> ChatGPT 和 Claude 的文件分析能力更强，但用户很难知道文件中哪些内容被模型使用、每一步花了多少 token、发生了哪些工具调用。我的项目重点解决透明性和边界控制：每一次上传、每一步工具调用、每一笔费用都有日志可查。

### 13.3 为什么不用完全本地模型

> 个人电脑很难稳定运行足够强的大模型。完全本地会牺牲结果质量。所以我选择混合架构：云端模型负责高质量推理，本地系统负责预处理、脱敏、权限控制、上传确认和执行审计。这样既保证智能效果，也保留用户对数据边界的控制权。

### 13.4 工程亮点

- 上传前预览
- 敏感信息脱敏
- 路径白名单
- token / cost 估算
- RunLog 可观测性
- 单次调用熔断
- 流式执行反馈
- pytest 覆盖安全策略

## 十五、最终结论

本项目最适合走“混合智能”路线：

- 智力来自云端大模型
- 文件控制留在本地
- 上传内容可预览
- 敏感信息可脱敏
- 成本消耗可估算
- 执行过程可审计

这样既避免了“完全本地但能力不足”的问题，也避免了“云端黑盒上传”的不安全感。

最终项目口号：

> 不是不用云端，而是让每一次云端调用都有边界、有记录、有选择权。
