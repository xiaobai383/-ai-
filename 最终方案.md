# 个人 AI 工作流助手 - 最终方案

## 一、最终定位

本项目不再定位为“完全本地、绝对隐私”的 AI 助手。这个方向虽然听起来安全，但在个人电脑无法运行高质量大模型的情况下，会牺牲智能效果，导致核心工作做不好。

最终定位调整为：

> 一个面向个人知识和工作文档的 AI 工作流助手，使用云端大模型保证智能效果，同时通过本地预处理、敏感信息脱敏、上传预览、执行日志和成本统计，让每一次 AI 调用都可控、可审计、可追踪。

一句话总结：

> 云端智力，本地边界。

本项目的差异化不再是“完全不上传云端”，而是“不会把用户文件黑盒式地丢给云端”。用户始终知道哪些内容会被发送、为什么发送、发送给哪个模型、预计消耗多少 token、最终花了多少钱。

## 二、核心差异化

如果只是做一个 LangChain Agent + Gradio + 几个 Python 工具，本项目很容易被 ChatGPT、Claude、Dify、Coze、LangFlow 等现成工具替代。

最终差异点应聚焦在两个方向：

### 2.1 可控上传

用户选择本地文件后，系统不是直接把全文发送给云端 LLM，而是先在本地完成：

- 文件解析
- 内容分块
- 关键信息抽取
- 敏感信息检测
- token 估算
- 成本预估
- 上传内容预览

然后由用户选择：

- 直接发送
- 脱敏后发送
- 只发送选中片段
- 取消任务

### 2.2 可审计 Agent

每次任务都生成完整 RunLog，记录：

- 用户原始请求
- 调用过哪些工具
- 每步输入和输出摘要
- 每步耗时
- 输入 token / 输出 token
- 估算费用
- 是否命中缓存
- 是否触发敏感信息拦截
- 最终结果保存路径

这让 Agent 从“黑盒聊天工具”变成“可追踪的工作流执行器”。

## 三、用户模式设计

不同任务对智能效果和隐私边界的要求不同，因此系统提供多种模式，而不是强行承诺一种绝对策略。

| 模式 | 适用场景 | 数据策略 |
|---|---|---|
| 快速模式 | 普通笔记、公开资料、低敏任务 | 直接调用云端 LLM，追求效果和速度 |
| 隐私增强模式 | 会议纪要、工作文档、合同初稿 | 本地预处理，只发送必要片段，可自动脱敏 |
| 手动确认模式 | 高敏感文件、客户资料、财务信息 | 发送前展示上传预览，用户确认后才发 |
| 本地兜底模式 | API 不可用或极敏感任务 | 只做本地解析、规则提取、模板生成，不承诺复杂推理质量 |

默认建议使用“隐私增强模式”，兼顾效果和边界。

## 四、核心流程

### 4.1 多轮对话流程（v1.1 主路径）

```text
进入对话界面
  ↓
新建 / 选择会话（模式选择：快速/隐私增强/手动确认/本地兜底）
  ↓
用户输入问题 + 可选上传文件
  ↓
文件走 RAG 流程（如有上传）：
  - 本地文件解析（txt / md）
  - ChromaDB 向量化与语义检索
  - 检索片段注入对话上下文
  ↓
历史上下文拼接（滑动窗口，默认保留最近 6 轮）
  ↓
敏感信息检测与脱敏（视模式而定）
  ↓
云端 LLM 流式推理 → 实时 token 消耗追踪
  ↓
回答逐 token 展示（打字机效果）
  ↓
完整消息持久化（SessionStore → JSON 文件）
  ↓
更新会话列表与余额显示
```

### 4.2 文件处理工作流路径（传统单轮）

```text
用户输入任务
  ↓
选择或拖入本地文件
  ↓
本地文件解析
  - txt / md / pdf / docx
  - 标题、段落、表格、页码
  ↓
敏感信息检测
  - 手机号 / 邮箱 / 身份证号 / API Key / 密码 / 绝对路径
  ↓
上传策略决策
  - 全文发送 / 片段发送 / 脱敏发送 / 用户手动选择
  ↓
上传预览（内容摘要、token 估算、费用预估、使用模型）
  ↓
用户确认
  ↓
云端 LLM 推理 / 本地模型兜底
  ↓
本地后处理 → 保存结果 → 展示 RunLog
```

## 六、技术选型

| 模块 | 实现技术 | 说明 |
|---|---|---|
| Agent 框架 | LangChain `create_agent` | Agent 编排 + 工具调用 |
| 多轮对话 | `src/agent/chat.py` 滑动窗口拼接 | 替换原有单轮 UI，支持会话历史上下翻 |
| 会话持久化 | SessionStore（JSON 文件） | 会话列表 + 消息按 session 分文件存储，重启不丢失 |
| 向量检索 | ChromaDB + Ollama nomic-embed-text | Cosine 距离 + 关键词加权排序 |
| 模型 | DeepSeek / OpenAI 兼容 API | 云端模型保证智能效果，支持实时余额查询 |
| 本地兜底 | Ollama（qwen2.5:1.5b） | 云端不可用时自动降级 |
| UI | Gradio（豆包式布局） | 左会话列表 + 右对话区，流式打字机效果 |
| API | FastAPI | RESTful 接口，方便桌面端/Web 前端接入 |
| 配置 | YAML + `.env` | 模型、目录白名单、费用阈值、模式配置文件化 |
| 日志 | RunLog（dataclass + JSONL） | 每步调用、token 消耗、费用记录 |
| 测试 | pytest + E2E 管线测试 | 工具、安全策略、成本估算、端到端流程 |

## 七、v0.1 必做功能

v0.1 不追求功能多，而要追求闭环完整、安全边界清晰、演示有说服力。

### 7.1 文件处理闭环

- 读取本地 txt / md 文件
- 支持 pdf / docx 可作为加分项
- 提炼摘要、待办、风险点
- 保存为 Markdown
- 返回保存路径

### 7.2 上传预览

发送云端前展示：

- 即将发送的内容片段
- 是否包含敏感信息
- 预计输入 token
- 预计费用
- 使用模型
- 用户确认按钮

### 7.3 敏感信息检测与脱敏

v0.1 先用规则检测即可：

- 手机号
- 邮箱
- 身份证号
- API Key 常见格式
- 密码字段
- 绝对路径

脱敏示例：

```text
张三 → PERSON_1
13812345678 → PHONE_1
admin@example.com → EMAIL_1
sk-xxxx → API_KEY_1
```

### 7.4 安全边界

- 路径白名单
- 文件大小限制
- 禁止读取 `.env`、私钥、系统配置等敏感文件
- 写文件前检查目标目录
- 覆盖已有文件前要求确认
- 单次工具调用次数上限
- 单次 token / cost 上限

### 7.5 RunLog

v0.1 必须实现结构化日志：

```python
from dataclasses import dataclass, field
from typing import List

@dataclass
class StepLog:
    step_id: int
    name: str
    input_preview: str
    output_preview: str
    duration_ms: int
    tokens_in: int = 0
    tokens_out: int = 0
    cost_yuan: float = 0.0
    status: str = "success"

@dataclass
class RunLog:
    run_id: str
    user_query: str
    mode: str
    model: str
    steps: List[StepLog] = field(default_factory=list)
    total_tokens_in: int = 0
    total_tokens_out: int = 0
    total_cost_yuan: float = 0.0
    result_path: str | None = None
```

### 7.6 流式反馈

Agent 执行多步骤任务时，界面必须持续反馈：

- 正在解析文件
- 正在检测敏感信息
- 等待用户确认上传
- 正在调用模型
- 正在保存结果
- 已完成

## 九、缓存策略

v0.1 只做轻量缓存，不引入本地 embedding 模型。

### 9.1 会话级缓存（v1.1）

SessionStore（JSON 文件）持久化所有历史消息，重启后自动恢复到上次会话，无需重复加载。

### 9.2 文件级缓存

基于文件路径、修改时间、文件大小生成缓存 key，避免重复读取与解析同一文件。

### 9.3 向量检索缓存（v1.0）

ChromaDB 持久化 embedding 向量，同一文件重复搜索无需重新嵌入。

### 9.4 暂不做语义缓存

语义缓存需要额外的 embedding 维护层，对于个人项目，文件级缓存 + 会话持久化 + 向量检索缓存已经足够。

## 十、成本控制

系统需要硬性熔断，避免 Agent 循环 bug 或大文件导致费用失控。

建议配置：

```yaml
limits:
  max_file_size_mb: 5
  max_tool_calls_per_request: 15
  max_tokens_per_request: 50000
  max_cost_per_request_yuan: 0.5
```

当预计费用超过阈值时：

1. 停止自动调用
2. 告知用户原因
3. 建议改为只发送部分片段
4. 允许用户手动确认继续

> v1.1 新增：每次对话后顶部状态栏显示实时余额（从 DeepSeek API 同步）。

## 十三、面试讲法

### 13.1 项目价值

> 我没有做一个简单的 ChatGPT 套壳，而是做了一个“可控上传、可审计执行”的个人工作流助手。它承认云端大模型的智能优势，但不会把用户文件黑盒式地直接丢给云端。系统会先本地解析、检测敏感信息、估算 token 和成本，再让用户确认发送范围。

### 13.2 和现成工具的区别

> ChatGPT 和 Claude 的文件分析能力更强，但用户很难知道文件中哪些内容被模型使用、每一步花了多少 token、发生了哪些工具调用。我的项目重点解决透明性和边界控制：每一次上传、每一步工具调用、每一笔费用都有日志可查。

### 13.3 为什么不用完全本地模型

> 个人电脑很难稳定运行足够强的大模型。完全本地会牺牲结果质量。所以我选择混合架构：云端模型负责高质量推理，本地系统负责预处理、脱敏、权限控制、上传确认和执行审计。这样既保证智能效果，也保留用户对数据边界的控制权。

### 13.4 工程亮点

- 上传前预览
- 敏感信息脱敏
- 路径白名单
- token / cost 估算
- RunLog 可观测性
- 单次调用熔断
- 流式执行反馈（打字机效果）
- 多轮对话会话持久化
- 知识库 RAG 注入
- DeepSeek 实时余额同步
- pytest + E2E 管线测试覆盖

## 十五、最终结论

本项目最适合走“混合智能”路线：

- 智力来自云端大模型
- 文件控制留在本地
- 上传内容可预览
- 敏感信息可脱敏
- 成本消耗可估算
- 执行过程可审计

这样既避免了“完全本地但能力不足”的问题，也避免了“云端黑盒上传”的不安全感。

最终项目口号：

> 不是不用云端，而是让每一次云端调用都有边界、有记录、有选择权。
